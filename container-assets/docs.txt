# Discobot Workspace Customization

Users can customize their workspace by adding files to `.discobot/` at the workspace root.

## .discobot/env

Optional workspace env file loaded into tools, console sessions, hooks, and services.
Changes are picked up for new tool executions, hook runs, service launches, and console sessions without restarting the session.

Format:
- One `KEY=VALUE` assignment per line
- `export KEY=VALUE` is also supported
- Blank lines and lines starting with `#` are ignored
- Quoted values are allowed and treated literally

Invalid lines are ignored with a warning that includes the file path and line number, but Discobot does not echo the rejected line contents.

## .discobot/scripts/

User-facing executable slash commands. Scripts live directly under
`.discobot/scripts/`: for example, `.discobot/scripts/summarize-diff.sh`
becomes `/summarize-diff`. Subdirectories are not searched.

Requirements:
- executable bit
- shebang line
- valid front matter
- file lives directly under `.discobot/scripts/`

Common front matter fields:
- `name`: slash-command name (defaults to normalized filename)
- `description`: human-readable description
- `visible`: whether the command is exposed to the agent; default `true`
- `argument-hint` / `argumentHint`: optional argument guidance

Behavior:
- users invoke them with `/name ...args`
- the agent invokes them through the `Skill` tool
- visible scripts are presented to the model the same way as skills
- hidden scripts are omitted from reminders and cannot be invoked by the model
- working directory is the project root (`/home/discobot/workspace`)
- Discobot passes everything after `/name` as a single raw string in `$1`
- success forwards trimmed `stdout`
- failure forwards execution metadata plus trimmed `stdout` and `stderr`
- empty successful output does not trigger a no-op model response

Discovery order for script-backed slash commands:
- workspace `.discobot/scripts/`
- user `~/.discobot/scripts/`
- user `~/.agents/scripts/`
- system `/opt/discobot/scripts/`
- system `/usr/local/share/discobot/scripts/`
- system `/usr/share/discobot/scripts/`

Example:

Path: `.discobot/scripts/summarize-diff.sh`

```bash
#!/bin/bash
#---
# name: summarize-diff
# description: Summarize the current git diff
# visible: true
#---
git diff --stat -- "$1"
```

## .discobot/hooks/

`.discobot/hooks/` supports two engines:
- `engine: script` (default): executable script with shebang and front matter.
- `engine: ai`: prompt file with front matter only. No executable bit or shebang
  is required. The body after front matter is the AI hook instruction prompt.

Requirements:
- all hooks: `type` field in front matter, file lives directly under `.discobot/hooks/`
- optional `phase: review` gates file hooks until the thread enters review
- script hooks: executable bit and shebang line
- AI hooks: `engine: ai`; `pre-commit` is not supported

Files run in alphabetical order — prefix with numbers for ordering.

### Front matter format

Use `#---` / `#---` delimiters with `# key: value` lines inside:

```bash
#!/bin/bash
#---
# name: Display Name
# type: session
#---
echo "script body"
```

Also supports `//---` (slash-prefixed) and plain `---` delimiters.

### Hook types

SESSION (type: session)
  Runs once at container startup before the agent starts.
  Fields: name, type, description, run_as (root|user, default: user)
  Env: DISCOBOT_SESSION_ID, DISCOBOT_WORKSPACE, DISCOBOT_HOOK_TYPE=session
  5-minute timeout. Failures are logged but don't block startup.

FILE (type: file)
  Runs after each LLM turn when matching files changed.
  Fields: name, type, description, phase (review only), pattern (required, glob),
  ignore/exclude (optional glob), notify_llm (default: true), engine (script|ai),
  subagent (AI only)
  Env for script hooks: DISCOBOT_CHANGED_FILES (space-separated paths), DISCOBOT_SESSION_ID, DISCOBOT_HOOK_TYPE=file
  Files matching a hook's ignore/exclude glob are skipped for that hook.
  Files matching .discobot/hooks/ignore are skipped for all file hooks.
  .discobot/hooks/ignore is plain text: one glob per line; blank lines and
  # comments are ignored. It uses the same glob syntax as pattern and does not
  support gitignore-specific features such as ! negation.
  If notify_llm is true and hook fails, the LLM is re-prompted (max 3 retries).
  If notify_llm is false, runs silently (useful for auto-fixers).
  With `phase: review`, matching changes are kept pending until the thread phase
  is `review`; the ReadyForReview tool sets that phase. Only `review` is valid.
  Glob uses picomatch: "*.go", "src/**/*.ts", "*.{ts,tsx}", "{package.json,pnpm*.yaml}"

AI HOOKS (engine: ai)
  AI hooks may be type: session or type: file. They are prompt files: the body
  after front matter tells the AI what to review. Each hook runs as a
  task-style review in a stable thread dedicated to that hook, and each run
  sends a new prompt with matching changed files and diff context. Active runs
  use the same task/completion tracking as Task and TaskOutput. Discobot also
  writes full context to:
    ~/.discobot/threads/{hookThreadId}/ai-hooks/{hookId}/context-{timestamp}.md
  The AI can read that file when inline context is truncated. Discobot adds a
  standard response instruction to each AI hook prompt: output SUCCESS when the
  hook passes, or FEEDBACK: <actionable feedback> when changes need attention.
  Hook authors should write review-specific instructions and do not need to
  include that success/feedback wording in the hook body. Optional subagent
  selects a configured session sub-agent. AI pre-commit hooks are not supported.

REVIEW PHASE (`phase: review`)
  File hooks may set `phase: review`; the parser currently accepts only `review` as
  a non-empty phase. Any hook with `phase: review` makes Discobot expose the
  ReadyForReview tool. File hooks with `phase: review` still match changed files
  after each LLM turn, but remain pending and do not execute until the thread
  phase becomes `review`. Calling ReadyForReview records the thread phase as
  `review`, making pending review-phase file hooks eligible on the next
  evaluation.

PRE-COMMIT (type: pre-commit)
  Installed as git pre-commit hook. Blocks commit on failure.
  Fields: name, type, description
  Existing .git/hooks/pre-commit is preserved as pre-commit.original.

### Examples

```bash
#!/bin/bash
#---
# name: Install system deps
# type: session
# run_as: root
#---
apt-get update && apt-get install -y postgresql-client
```

```bash
#!/bin/bash
#---
# name: Go format check
# type: file
# pattern: "*.go"
# ignore: "*_templ.go"
#---
gofmt -l $DISCOBOT_CHANGED_FILES
```

```bash
#!/bin/bash
#---
# name: ESLint autofix
# type: file
# pattern: "*.{ts,tsx}"
# notify_llm: false
#---
npx eslint --fix $DISCOBOT_CHANGED_FILES
```

```markdown
---
name: Go reviewer
type: file
engine: ai
pattern: "**/*.go"
phase: review
subagent: reviewer
---
Review changed Go files for correctness, missing tests, and maintainability.
```

```bash
#!/bin/bash
#---
# name: CI
# type: pre-commit
#---
pnpm run ci
```

## .discobot/services/

Background processes with optional HTTP proxying.

### Executable services

Executable scripts that Discobot starts/stops as child processes.
Requirements: executable bit, shebang line, script body.
Output is stored at `~/.discobot/services/{service-id}/output.log` as JSONL (max 1MB, auto-truncated).

Fields: name, description, order (lower numbers appear first in the UI), http (port number), https (port number), path (URL path for preview)

Service ID is derived from filename: extensions stripped (.sh,.py,.js,etc), lowercase, non-alphanumeric removed.
Services with `order` sort before unordered services. Ties fall back to alphabetical name order.

```bash
#!/bin/bash
#---
# name: Dev Server
# description: Vite development server
# order: 10
# http: 3000
#---
pnpm dev
```

```bash
#!/bin/bash
#---
# name: SQLite GUI
# description: Database browser
# http: 8080
#---
exec uvx --from sqlite-web sqlite_web "$HOME/app.db" --port 8080 --host 0.0.0.0 --no-browser
```

### Passive services

Declare an HTTP port for an externally-managed service (no script body needed).
Does not need executable bit or shebang. Must have http or https and empty body.

```yaml
---
name: Web App
description: Started by devcontainer
order: 10
http: 3000
---
```

### HTTP proxy

Services with http/https get automatic proxying via subdomain URLs:
  {session-id}-svc-{service-id}.{base-domain}
Non-passive services auto-start on first HTTP request.
The `path` field sets the default URL path in the web preview.
